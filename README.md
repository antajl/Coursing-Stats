# ProCoursing Stats — спецификация проекта

## Цель

Агрегатор статистики по собакам, участвовавшим в состязаниях procoursing.ru
(курсинг, БЗМП, бега/трасса) за все годы (2015–2026). По каждой собаке —
история выступлений, баллы, титулы. Топы/рейтинги с фильтрами по породе и году.

## Использование парсеров

### Установка зависимостей
```bash
npm install
```

### Скрапинг индекса событий
```bash
npm run scrape-index
```
Создаёт `events.json` со списком всех событий за 2015-2026.

### Парсинг результатов курсинга
```bash
npm run parse-coursing <url>
# Пример:
npm run parse-coursing http://procoursing.ru/2026/2026-04-04_Complete_Results_Coursing.html
```

### Парсинг результатов БЗМП
```bash
npm run parse-bzmp <url>
# Пример:
npm run parse-bzmp http://procoursing.ru/2026/2026-04-19_Complete_Results_BZMP.html
```

### Парсинг результатов бега
```bash
npm run parse-racing <url>
# Пример:
npm run parse-racing http://procoursing.ru/2026/2026-05-16_Complete_Results_Racing.html
```

## Источник данных

`http://procoursing.ru/s_{YEAR}.html` — индекс событий за год (2015…2026).
Каждая строка таблицы — событие:

- Дата (бывает диапазон, напр. `18-19.04.2026`)
- Тип/ранг (ЧРКФ, CACL, РКФ…) + подтип (Курсинг / БЗМП / Бега)
- Название состязания
- Принимающий клуб
- Регион/город
- Ссылка на каталог (PDF)
- Ссылка на результаты (HTML)

Страница результатов — HTML-таблица с заголовками групп пород и строками собак.

⚠️ Кодировка сайта — **windows-1251**. Декодируется явно через iconv-lite.

## Архитектура (по аналогии с SW Forge)

```
Источник (procoursing.ru)
   │  Cron Trigger (Cloudflare Worker, раз в день/неделю)
   ▼
Скрапер/парсер (Node, локально для бэкафилла; Worker — для инкремента)
   │
   ▼
Cloudflare D1 (events, dogs, results)
   │
   ▼
Pages-сайт (фронтенд: профиль собаки, топы, фильтры)
```

Бэкафилл 2015–2025 — разовый прогон скрипта локально (с задержкой между
запросами, не агрессивно). Текущий год — инкрементальное обновление по крону.

## Схема БД

См. `schema.sql`. Ключевые решения:

- `results.raw_scores_json` — судейские баллы по кругам храним как JSON,
  потому что набор полей разный для Coursing/БЗМП/Бега. Вычисляемые
  `total_score` нормализуем отдельно для сравнимости между типами событий.
- Идентификация собаки — по `(name_lat, breed)` как временный ключ.
  Это ненадёжно (опечатки, разное написание). Если парсить PDF-каталоги —
  там есть номер родословной/клеймо, что надёжнее. Это можно сделать
  фазой 2; на фазе 1 предусмотрена ручная склейка дублей (поле
  `merged_into_dog_id`).

## Топы (зафиксировано)

Два отдельных топа, не одна формула — потому что "1 победа из 1 старта" и
"5 стартов без побед" нельзя честно сравнить одним числом:

- **Топ по местам** — медальный зачёт (как в спорте): сортировка по числу
  1-х мест, потом 2-х, потом 3-х. Решает проблему сравнения само по себе.
- **Топ по очкам** — таблица с лучшим результатом, средним баллом и числом
  стартов рядом — зритель сам видит контекст, сортировка по любой колонке.
- В UI обоих топов — необязательный фильтр "минимум N стартов" (по
  умолчанию выключен, это выбор зрителя, а не наш).

Базовая (без фильтров) логика — `v_top_by_placement` / `v_top_by_score` в
`schema.sql`. Проверено на реальном SQLite.

## API эндпоинты (для фронтенда)

Фронтенд обращается не прямо к D1, а к Cloudflare Worker, который оборачивает
view'ы динамическими фильтрами (`WHERE`/`JOIN events`, которых сами view не
принимают):

- `GET /api/top/placement?breed=&year=` — медальный зачёт (курсинг + БЗМП)
- `GET /api/top/score?breed=&year=` — топ по очкам (курсинг + БЗМП)
- `GET /api/dogs/:id` — профиль собаки с раздельной статистикой:
  - Базовая информация: name_lat, name_ru, breed, sex, owner
  - `coursing_stats`: total_starts, best_score, avg_score, gold, silver, bronze
  - `racing_stats`: total_starts, best_speed, avg_speed
- `GET /api/breeds` — список пород для дропдауна фильтра (`SELECT DISTINCT breed FROM dogs`)
- `GET /api/years` — список годов для фильтра (`SELECT DISTINCT year FROM events`)
- `GET /api/events?year=` — список событий с фильтром по году

## Развертывание

### GitHub
1. Создать репозиторий на GitHub
2. Залить файлы проекта
3. Подключить к Cloudflare Pages

### Cloudflare
1. Создать D1 базу данных
2. Выполнить `schema.sql` для создания таблиц
3. Создать Cloudflare Worker для API
4. Настроить Cron Trigger для инкрементального обновления
5. Подключить GitHub к Cloudflare Pages для деплоя фронтенда

### Локально
Для локального тестирования можно:
- Запускать парсеры напрямую через npm scripts
- Использовать локальный SQLite для разработки (аналог D1)

## Открытые вопросы / риски

- Объём: на 2026 ~50 событий → 12 лет ≈ 300–400 событий.
- Скрапинг с задержкой 1-2 сек между запросами (вежливо к сайту).
- Опечатки в кличках — ручная склейка дублей через админку.
