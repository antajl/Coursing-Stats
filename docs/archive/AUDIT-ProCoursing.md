# Аудит ProCoursing Stats — почему изменения даются тяжело и что с этим делать

Прочитал `docs/`, схему, `worker.js` целиком, парсеры, фронтенд-сервис API, git-историю
(28 коммитов). Ниже — что нашёл, по приоритету. Везде указаны конкретные файлы/строки,
чтобы следующий ИИ-агент мог проверить сам, а не верить мне на слово.

**Итог сразу:** переписывать всё с нуля — не нужно. Модель данных (events/dogs/results +
view'ы топов) спроектирована правильно и держится уже на 4639 результатах. Проблема не в
фундаменте, а в том, как накопился код вокруг него: один разросшийся файл API, ноль
автотестов и несколько незакрытых "хвостов" от экспериментов, которые забыли убрать после
ревёртов. Это объясняет и "много действий на маленькое изменение", и ощущение "суть работает
не так хорошо" — у тебя реально есть баг, который тихо портит сам рейтинг.

## 1. Критичные находки

### 1.1. 🔴 Баг шкалы баллов — рейтинг "по очкам" сравнивает несравнимое

`backend/parsers/parse-results-coursing.mjs`:

- Формат 2025-2026 (25 ячеек): `totalScore = grandTotal` — **сырая сумма** баллов судей
  (строка 391-392, комментарий "без деления"). Это та шкала ~0-400, про которую ты говорил.
- Формат 2023-2024 (23 ячейки, "simplified"): `totalScore = rawTotalScore / judgeCount`
  (строка 405-406) — **средний балл на одного судью**, число совсем другого порядка.

Оба значения пишутся в одну и ту же колонку `results.total_score`, и `v_top_by_score` /
`/api/top/score` без фильтра по году молотят `MAX()`/`AVG()` по ней вперемешку. Получается:
собака, выступавшая в 2023-м, при объективно той же результативности окажется в рейтинге
ниже собаки 2025-2026 года — просто из-за разных единиц измерения, а не реальной разницы в
качестве. Это бьёт прямо в смысл сайта ("кто самая крутая собака").

**Что делать:** выбрать одну шкалу (логичнее — сырая сумма, она у тебя уже эталон для
2025-2026 и совпадает с твоим "максимум 400"), пересчитать существующие 2023-2024 записи
бэкафиллом, и держать инвариант "total_score всегда в одной шкале" как зафиксированное
правило в `docs/01-AI-GUIDELINES.md` — иначе при следующем формате (что если 2027-й опять
будет другим?) баг повторится.

**✅ ВЫПОЛНЕНО (с корректировкой):** После уточнения от пользователя (главный судья тоже считает, бывает 1-3 судьи) понял что сырая сумма не сравнима. Сделал нормализацию по количеству судей (grand_total / judge_count) вместо сырой суммы. Это правильнее, но отличается от исходной рекомендации.
- Добавлена функция `extractJudgeCount()` в парсер
- Парсер теперь делит на реальное количество судей
- Создана SQL миграция `data/migrate-normalize-total-score.sql`
- Правило добавлено в `docs/01-AI-GUIDELINES.md`

### 1.2. 🔴 Незащищённые admin-эндпоинты, один торчит кнопкой на сайте

В `backend/src/worker.js` без какой-либо авторизации:
- `POST /api/recreate-views` (строка 114) — дропает и пересоздаёт view'ы в продовой БД
- `POST /api/admin/reparse-coursing` (строка 150) — запускает репарсинг событий
- `POST /api/update/trigger` (строка 135) — сейчас просто заглушка, но по комментарию в коде
  задуман как реальный триггер обновления

Любой человек в интернете может curl'ом дёрнуть любой из них. Хуже — третий буквально
выведен кнопкой "Обновить данные" в навбаре сайта для всех посетителей
(`frontend/src/App.jsx`, строки 54-62). Сейчас она безопасна (эндпоинт ничего не делает), но
если её когда-нибудь доведут до реальной логики — это открытый кран на скрапинг
procoursing.ru по требованию любого анонимного визитора, что прямо противоречит твоему же
правилу "будь вежлив к сайту-источнику" из AI-GUIDELINES.

**Что делать:** добавить простую проверку секретного заголовка (`X-Admin-Token`, сравнение с
переменной окружения Worker'а) на все `/api/admin/*` и `/api/recreate-views`, `/api/update/*`.
Кнопку из публичного навбара убрать — это служебное действие, не пользовательская фича.

**✅ ВЫПОЛНЕНО:**
- Добавлена функция `checkAdminToken()` в worker.js
- Все admin-эндпоинты теперь требуют заголовок X-Admin-Token
- Кнопка "Обновить данные" удалена из App.jsx
- Удалены неиспользуемые useState и handleUpdate

### 1.3. 🟡 Мёртвый код — три случая

1. **Дублирующийся роут.** `/api/dogs/:id/events` определён дважды в `worker.js` (строка 351
   и строка 510, идентичный regex). Так как это цепочка `if`, второй блок (с полем
   `competition_kind`) никогда не выполняется — мёртвый код, который выглядит как рабочий.
2. **Целая теневая копия API.** `frontend/functions/api/[[path]].js` (322 строки) — это
   полная переимплементация всего API для Cloudflare Pages Functions, оставшаяся от
   эксперимента "избежать CORS через Pages Functions" (коммиты `63ec470` → `de8bcbf`, 7
   коммитов туда-обратно, закончившиеся ревёртом). Сейчас `frontend/src/services/api.js`
   ходит прямо в Worker по `API_URL`, этот файл не используется вообще, но 322 строки с
   другой (устаревшей) версией бизнес-логики продолжают лежать в репозитории и могут сбить с
   толку следующего агента, который не в курсе истории.
3. **Двойной cron.** В `wrangler.toml` у Worker'а триггер по понедельникам 02:00 UTC, и
   `scheduled()` в `worker.js` (строки 32-58) — но это просто `console.log`, реальной логики
   нет ("For now, just log the intent"). Настоящее обновление делает GitHub Actions
   (`update-db.yml`) часом позже, в 03:00 UTC. То есть в конфиге заявлена одна схема, а
   реально работает другая — путаница для любого, кто читает `wrangler.toml` впервые.

**Что делать:** удалить `frontend/functions/`, удалить вторую копию роута, либо реализовать
`scheduled()` по-настоящему, либо убрать `[triggers]` из `wrangler.toml` и оставить
GitHub Actions единственным источником обновлений (он уже и так работает — см. `00-OVERVIEW.md`).

**✅ ВЫПОЛНЕНО:**
- Удалена папка `frontend/functions/` полностью
- Удалён дублирующийся роут `/api/dogs/:id/events` (вторая копия)
- ✅ Двойной cron: удалён [triggers] из wrangler.toml и scheduled() из worker.js (GitHub Actions используется для обновлений)

## 2. Почему "даже маленькое изменение — много действий" (структурная причина)

Это не одна причина, а сумма нескольких:

- **Нет автотестов вообще**, кроме одного теста парсера на синтетических данных
  (`backend/scripts/test-parser.mjs`). Ноль тестов для Worker API, ноль для фронтенда. Когда
  тестов нет, единственный способ агента убедиться, что правка не сломала что-то — поднять
  оба сервера руками и проверить глазами. Это и есть "много действий" — не потому что задача
  сложная, а потому что верификация дорогая. **Это самая дешёвая по эффекту правка из всех.**
- **`worker.js` — монолит на 598 строк**: один файл = роутинг + CORS + SQL-запросы +
  бизнес-логика + admin-эндпоинты в одной цепочке `if`. Любое изменение одного эндпоинта
  требует прочитать/удержать в контексте весь файл, повышая риск зацепить соседний роут (как
  в п. 1.3 — дублирование родилось именно от этого).
- **`backend/scripts/` — 25 файлов без разделения "ещё нужны" / "уже отработали и не нужны"**:
  `backfill-2023.mjs`, `backfill-2024.mjs`, `backfill-2025.mjs`, `backfill-2026.mjs`,
  `backfill-historical.mjs`, `migrate-*` (3 шт.), `reparse-*` (3 шт.), `update-*` (6 шт.) — все
  вперемешку с теми 6, что реально используются регулярно (`scrape-year-index`,
  `load-events`, `load-results`, `ci-update-db`, `sync-local-to-remote`, `test-parser`).
  Агенту (и тебе) приходится каждый раз заново разбираться, какой скрипт ещё актуален.

**✅ ВЫПОЛНЕНО:**
- Создана папка `backend/scripts/archive/`
- Перемещены в archive/: backfill-2023/2024/2025/2026/historical.mjs, migrate-add-judge-count.mjs, migrate-normalize-existing-scores.mjs
- Перемещены в archive/ (по рекомендации из раздела 6): reparse-coursing-events.mjs, reparse-standalone.mjs, run-reparse-direct.mjs, update-bzmp-status-reasons.mjs, update-judges.mjs, update-missing-bib-numbers.mjs, update-raw-scores-simple.mjs, update-raw-scores.mjs, update-status-reasons.mjs, update-track-schemes.mjs, check-missing-results.mjs
- Остались в scripts/: ci-update-db.mjs, load-events.mjs, load-results.mjs, merge-dogs.mjs, migrate-normalize-dog-names.mjs, scrape-year-index.mjs, sync-local-to-remote.mjs, test-parser.mjs, update-current-year.mjs
- **Документация продублирована.** Корневой `README.md` и `docs/00-OVERVIEW.md` описывают
  одно и то же (статус, что сделано, ограничения) — два источника правды, которые легко
  разойдутся при следующем изменении (один обновят, другой забудут).

**✅ ВЫПОЛНЕНО:**
- README.md сокращён до короткого описания со ссылками на docs/
- docs/00-OVERVIEW.md остаётся основным источником правды

- **Mock-данные в проде.** `frontend/src/data/mockData.js` (249 строк) зашиты в продовый
  бандл, и при таймауте/ошибке API (`frontend/src/services/api.js`, `fetchWithFallback`,
  таймаут 5 сек) сайт молча показывает их вместо реальных данных — да, есть баннер "Используются
  демо-данные" в `TopDogs.jsx`, это не полностью скрыто, но для статистического сайта странно,
  что при простой просадке скорости API посетитель может увидеть выдуманный рейтинг собак,
  выглядящий как настоящий.

**✅ ВЫПОЛНЕНО (с корректировкой):** После жалобы пользователя на фейковых собак в рейтинге и событие не найдено, понял что полностью убрать fallback было ошибкой. Вернул `fetchWithFallback()` для продакшена, но с явным предупреждением в UI когда используются mock-данные.
- Возвращён `fetchWithFallback()` для всех API вызовов в продакшене
- Mock-данные для локальной разработки (import.meta.env.DEV)
- Добавлен баннер с предупреждением когда используются демо-данные
- **Пустая `.agents/` папка** в корне и название пакета в `package.json` — всё ещё
  `"procoursing-stats-scraper"`, хотя проект давно вырос за рамки скрапера — мелочи, но
  показывают, что базовые вещи не пересматриваются по мере роста проекта.

## 5. Структура файлов для удобства AI-агентов

Принцип: маленькие файлы с одной задачей > большие файлы с десятком задач — агенту дешевле
прочитать и безопаснее редактировать. Конкретно для этого проекта:

```
backend/
  src/
    worker.js          # только диспетчер + CORS, ~30 строк
    routes/
      events.js
      dogs.js
      top.js
      admin.js         # с авторизацией, отдельно от остального — видно, что это чувствительная зона
  scripts/             # только то, что реально используется регулярно
  scripts/archive/     # одноразовые backfill/migrate/reparse — пометить как "не запускать повторно"
  tests/
    fixtures/          # реальные HTML-снэпшоты по форматам/годам
    worker.test.mjs
    parser.test.mjs
docs/
  00-OVERVIEW.md
  01-AI-GUIDELINES.md
  02-ARCHITECTURE.md
  03-DEVELOPMENT.md
  04-DECISIONS-LOG.md  # новый: короткие датированные записи "пробовали X → не сработало
                        # из-за Y → не повторять" (сразу закрыло бы повтор истории с
                        # Pages Functions из 7 коммитов)
```

`04-DECISIONS-LOG.md` — отдельный совет: история коммитов показывает, что Pages Functions
уже пробовали и откатили (7 коммитов туда-обратно). Без письменной заметки об этом следующий
агент (или ты сам через полгода) может предложить то же самое снова и повторить тот же путь.
Одна строчка в логе решений стоит дешевле, чем повторный эксперимент.

✅ ВЫПОЛНЕНО: Создан файл docs/04-DECISIONS-LOG.md с записями о:
- Pages Functions эксперимент

## 6. Принятые решения (зафиксировано Antal'ом 26.06.2026)

1. **Шкала `total_score`** — эталон: сырая сумма (как в 2025-2026, максимум ~400).
   2023-2024 переводятся миграцией по существующим данным, без повторного скрапинга — нужное
   значение уже лежит в `raw_scores_json.raw_total`:
   ```sql
   UPDATE results SET total_score = json_extract(raw_scores_json, '$.raw_total')
   WHERE json_extract(raw_scores_json, '$.format') = 'simplified_2024'
     AND raw_scores_json IS NOT NULL;
   ```
   Плюс правка в `parse-results-coursing.mjs` (~строка 406): для будущих реимпортов не делить
   на `judgeCount`, использовать `rawTotalScore` напрямую.

**⚠️ КОРРЕКТИРОВКА:** После уточнения от пользователя (главный судья тоже считает, бывает 1-3 судьи) понял что сырая сумма не сравнима. Сделал нормализацию по количеству судей (grand_total / judge_count) вместо сырой суммы. Это правильнее, но отличается от исходной рекомендации.

2. **Кнопка "Обновить данные"** — убрать из публичного навбара (`App.jsx`) полностью. Эндпоинт
   `/api/update/trigger` закрыть авторизацией независимо от UI — это вопрос безопасности API.

✅ ВЫПОЛНЕНО.

3. **`frontend/functions/[[path]].js`** — удалить. Перед удалением: `grep -rn "functions/api"`
   по репозиторию, чтобы убедиться, что ничего на него не ссылается, затем удалить файл.

✅ ВЫПОЛНЕНО.

4. **`backend/scripts/`** — разложить:
   - **Оставить в `scripts/`** (активно используются): `scrape-year-index.mjs`,
     `load-events.mjs`, `load-results.mjs`, `ci-update-db.mjs`, `update-current-year.mjs`
     (вызывается из `ci-update-db.mjs` через `execSync`, хотя не прописан в `package.json` —
