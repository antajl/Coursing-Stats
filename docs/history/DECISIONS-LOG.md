## 2026-06-26: Покупка домена для решения проблемы uBlock Origin

**Проблема:**
- uBlock Origin блокирует cross-origin запросы к `antajltube.workers.dev` как потенциальный трекер
- Это edge case одного расширения, но uBlock популярен среди пользователей
- Pages Functions эксперимент уже пробовали и откатили (задокументировано ниже)

**Решение:**
- Куплен домен `antajl.ru` (основной) и `antajl.online` (дополнительный)
- NS записи изменены на Cloudflare (raquel.ns.cloudflare.com, rohin.ns.cloudflare.com)
- Настроены поддомены в Cloudflare:
  - `procoursing.antajl.ru` → Pages (фронтенд)
  - `procoursing-stats.antajltube.workers.dev` → Worker (API)
- После обновления DNS фронтенд будет делать запросы к собственному домену, uBlock не будет блокировать

**Реализация:**
- Домены куплены на Reg.ru за 170 рублей
- NS записи изменены на Cloudflare
- Поддомены настроены в Cloudflare Dashboard
- Ожидание обновления DNS (1-2 часа, до 24 часов)

**Коммиты:**
- Обновлена документация (00-OVERVIEW.md) с информацией о новых доменах

## 2026-06-29: Исправление ошибок после анализа кода

**Найдено и исправлено 5 проблем в результате ревью ключевых файлов:**

### 1. Старый URL API в продакшене
- **Файл:** `frontend/src/services/api.ts`
- **Проблема:** Хардкод `https://procoursing-stats.antajltube.workers.dev` — старый Worker URL. Продакшн домен уже `https://api.procoursing.antajl.ru`
- **Исправление:** Заменён на `https://api.procoursing.antajl.ru`
- **Риск был:** Все API запросы в продакшне шли на старый адрес если `VITE_API_URL` не задан в Cloudflare Pages

### 2. Сломанный импорт в admin.ts
- **Файл:** `backend/src/routes/admin.ts`
- **Проблема:** `POST /api/admin/reparse-coursing` пытался сделать `import('../scripts/reparse-coursing-events.mjs')` — файл давно перемещён в `archive/` и не экспортирует нужную функцию
- **Исправление:** Эндпоинт возвращает 501 с понятным сообщением "Use: npm run reparse-coursing"

### 3. Фильтр breed в /api/competitions не работал
- **Файл:** `backend/src/routes/events.ts`
- **Проблема:** Параметр `breed` читался из запроса, но не добавлялся в SQL WHERE
- **Исправление:** Добавлен подзапрос `AND id IN (SELECT DISTINCT r.event_id FROM results r JOIN dogs d ...)`

### 4. ELSE 0 вместо ELSE NULL в top/speed
- **Файл:** `backend/src/routes/top.ts`
- **Проблема:** В `MAX(CASE ...)` для скорости стояло `ELSE 0` — собаки без данных о скорости попадали в рейтинг со скоростью 0
- **Исправление:** `ELSE NULL` — такие записи корректно исключаются из `MAX()`

### 5. Схемы Zod не совпадали с реальными API ответами
- **Файл:** `frontend/src/schemas/api.ts`
- **Проблемы:**
  - `TopPlacementResponseSchema`: ожидал `total_wins`, `win_rate` — API возвращает `gold`, `silver`, `bronze`
  - `TopScoreResponseSchema`: ожидал `max_score` — API возвращает `best_score`
  - `DogProfileResponseSchema`: совершенно другая структура (не было `coursing_stats`, `racing_stats`)
  - `DogEventsResponseSchema`: ожидал `{ event, result }` — API возвращает плоский массив
  - `BreedsResponseSchema`: ожидал `z.array(z.string())` — API возвращает `[{ breed: string }]`
  - `YearsResponseSchema`: ожидал `z.array(z.number())` — API возвращает `[{ year: number }]`
- **Исправление:** Все схемы приведены в соответствие с реальными ответами API
- **Примечание:** Схемы импортируются но нигде не вызываются через `.parse()` — runtime валидация остаётся в планах (см. `10-FUTURE-PLANS.md`)
