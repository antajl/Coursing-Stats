# Документация по рекордам Донино

> **ИИ:** карта задач — [README.md](README.md). Файлы данных — [03-DATA.md](03-DATA.md) (`donino/`).

## Два вида замеров (не пересекаются)

На полигоне Донино ведутся **две независимые серии замеров**. Это разные дисциплины, разные Google-таблицы, разные таблицы в D1 и **разная статистика на сайте**. Данные одной серии **нельзя** подставлять в другую (в т.ч. пересчитывать время из км/ч замера скорости и наоборот).

### 1. Замер скорости

**Что это:** на собаку надевают ошейник с датчиком; во время **курсинга** (бег за мехом по трассе) фиксируют **скорость в км/ч**. Это не отдельный забег на ровные 350 м с секундомером у финиша.

| | |
|---|---|
| **Смысл** | Личный рекорд скорости на курсинге |
| **Первичная величина** | км/ч (из таблицы / прибора) |
| **Google Sheet** | [1NTiY3HXZ…](https://docs.google.com/spreadsheets/d/1NTiY3HXZIkXE8xTeXZESgMKaZsEXunmcWhTfhhkoKyE) |
| **D1** | `speed_records` |
| **API** | `GET /api/speed-records` |
| **UI** | Переключатель **Записи | Статистика** (`?view=stats`) на `/speed-records`; колонка «Замер» в двухколоночном layout |

### 2. Бега борзых 350 м

**Что это:** отдельная **ровная трасса ровно 350 метров**. Собаку пробегают по ней, время засекают **секундомером**. Это не замер ошейником на курсинге.

| | |
|---|---|
| **Смысл** | Зачёт на фиксированной дистанции 350 м |
| **Первичная величина** | секунды (колонка «Время» в таблице) |
| **Google Sheet** | [1hpdA8vl…](https://docs.google.com/spreadsheets/d/1hpdA8vlIfeECgpnPvuk5xfezPsdUh1EXULjeATAF9dw) |
| **D1** | `coursing_records` |
| **API** | `GET /api/coursing-records` |
| **UI** | Та же страница, колонка «Бега 350 м»: время (сек), карточки групп в статистике |

**Скорость км/ч в статистике 350 м** — только **вспомогательный пересчёт** для сравнения: `1260 / время_сек`. В таблице курсинга её нет как исходного поля.

### Одна собака в обеих дисциплинах

Одна и та же собака (**кличка + порода**) может участвовать **и в бегах 350 м, и в замере скорости на курсинге**. Это одна собака, не дубликат в данных.

| | Записи в `speed_records` | Записи в `coursing_records` |
|---|--------------------------|-----------------------------|
| Что это | замеры км/ч с ошейника | зачёты сек на трассе 350 м |
| Связь | `name` + `breed` (+ `sex` в листе) | `name` + `breed` |
| На сайте | блок «Замер скорости» в профиле Донино | блок «Бега 350 м» в профиле Донино |

**Метрики не объединяются:** лучший км/ч с курсинга и лучшие секунды на 350 м — два разных рекорда. На сайте — **два независимых режима статистики** внутри соответствующей дисциплины.

### Правила для кода и UI

1. **Не смешивать источники:** `speed_records` ↔ `coursing_records` только на уровне профиля собаки (две отдельные карточки), не в одной агрегированной формуле «лучшее время».
2. **Не выводить время 350 м из max(км/ч)** таблицы замеров — это была типичная ошибка (напр. 19,69 с из 64 км/ч вместо 22,81 с из реального зачёта).
3. **Фильтры статистики:** в режиме «Статистика» у бегов 350 м — только `coursing_records`; у замера скорости — только `speed_records`. Фильтры не общие с таблицей записей.
4. **Пол** в статистике бегов 350 м подтягивается из `speed_records` по `name+breed` (в листе курсинга пола нет) — единственное допустимое «пересечение» таблиц, **только для группировки**, не для слияния метрик.
5. **Одна собака в обоих листах** — нормально; в UI показывать два независимых блока, не один сводный рекорд.

## Обзор (техническая сводка)

Данные Донино — **два отдельных источника** в Google Sheets, две таблицы в D1:

| Назначение | Google Sheets | Таблица D1 | Единица |
|------------|---------------|------------|---------|
| **Замер скорости** | [1NTiY3HXZ…](https://docs.google.com/spreadsheets/d/1NTiY3HXZIkXE8xTeXZESgMKaZsEXunmcWhTfhhkoKyE) | `speed_records` | км/ч |
| **Бега борзых 350 м** | [1hpdA8vl…](https://docs.google.com/spreadsheets/d/1hpdA8vlIfeECgpnPvuk5xfezPsdUh1EXULjeATAF9dw) | `coursing_records` | сек |

**Не смешивать:** лучшее время 350 м из `speed_records` (пересчёт `1260 / км/ч`) и реальный зачёт с трассы 350 м — разные числа. Статистика 350 м на сайте берётся только из `coursing_records`.

Система использует гибридный подход:
1. **Google Sheets** — основной источник
2. **Локальный JSON кэш** — `data/speed-records.json` (legacy; для контроля версий)
3. **data/v1/** — `donino/speed_records.json` (runtime; попадает в `pc-db.sqlite` и на Pages)
4. **База данных D1** — импорт/cron (не prod runtime)
5. **Скрипты синхронизации** — парсинг XLSX → SQL → D1 → export в data/v1

### Парсинг листа 350 м

На листе **одна строка на собаку** (не несколько строк на кличку). История прежних рекордов хранится не в отдельных строках, а в **примечании ячейки колонки «Дата»** (при наведении в Google Sheets).

- **Парсер**: `backend/scripts/speed/parse-coursing-xlsx.ts` — XLSX с `cellStyles: true`, абсолютные координаты колонок
- **Статусы и примечания**: `backend/scripts/speed/coursing-sheet-status.ts`
- **Формат примечания**: строки вида `23,36 - 31.05.2026` (время с запятой, дата `DD.MM.YYYY`) → JSON `history: [{ time_seconds, date }, …]`
- **Анализ цветов/комментариев**: `npx tsx backend/scripts/speed/analyze-coursing-xlsx.ts`

| Признак | Интерпретация в парсере |
|---------|-------------------------|
| Примечание в «Дата» | `history` из текста; `status` = `improved`; бейдж `upd` в UI |
| Белая заливка `#FFFFFF`, без примечания | `status` = `new` (новая запись на листе) |
| Серая `#CCCCCC`, без примечания | `status` = `normal` (большинство строк) |

**Про серый цвет:** по данным XLSX (июль 2026) `#CCCCCC` — **фон по умолчанию** у ~79 из 107 строк, а не отдельный статус как зелёный/синий в листе замера скорости. Улучшенные рекорды (`improved`) встречаются **и на сером, и на белом** фоне — различие, вероятно, «давно обновлено» vs «недавно подсвечено белым», но это **не подтверждено** владельцем таблицы. Для UI достаточно примечания: есть `history` → `upd`.

Пример **Драми**: в строке `23,03` от `28.06.2026`, в примечании `23,36 - 31.05.2026` и `23,19 - 06.06.2026` → `history` из двух прошлых результатов.

## Пайплайн обработки данных (speed)

### 1. Скрипт синхронизации
- Скрипт: `backend/scripts/speed/sync-speed-records.ts`
- Парсинг: `backend/scripts/speed/parse-speed-xlsx.ts` — чтение XLSX по **абсолютным координатам** листа (`cellStyles: true`), цвет заливки колонки «Кличка»
- Статусы из цвета: `backend/scripts/speed/speed-sheet-status.ts` — `#B6D7A8` → `new`, `#9FC5E8` → `improved`, без цвета → `normal`; лист «старые личные рекорды» без цвета → `old`
- Загружает XLSX файл из Google Sheets
- Парсит все листы и извлекает записи
- Сравнивает с локальным кэшем (`data/speed-records.json`)
- Идентифицирует новые и обновлённые записи
- Обновляет локальный кэш
- Генерирует SQL и загружает в базу данных D1

### 2. Дедупликация
- **Цель**: Удалить дублирующиеся записи, которые могут существовать в разных листах
- **Ключ дедупликации**: `name + breed + sex + date + speed_km_h`
- **Пример**: Если та же запись (та же собака, та же дата, та же скорость) появляется в обоих листах "2025" и "2025 по породам", остаётся только одна
- **Результат**: 418 записей → 187 уникальных записей (231 дубликат удалён)

### 3. Сравнение кэша
- **Цель**: Идентифицировать новые и обновлённые записи
- **Ключ сравнения**: `name + breed + sex + date + speed_km_h`
- **Вывод**: Показывает количество новых и обновлённых записей
- **Действие**: Обновляет файл локального кэша со всеми записями

### 4. Группировка для истории
- **Ключ группировки**: `name + breed`
- **Цель**: Группировать записи по собакам для формирования истории
- **Логика**: Собаки с одинаковым именем и породой считаются одной и той же собакой, независимо от пола
- **Пример**: "Тайга" + "Салюки" → все записи для этой собаки группируются вместе

### 5. Формирование истории
- Для каждой собаки записи сортируются по дате (от новых к старым)
- Для каждой записи история содержит все другие записи для этой собаки (кроме текущей)
- Формат истории: `[{speed_km_h, date}, ...]`
- **Пример**: Если у Тайги 3 записи (55 км/ч 11.05.2026, 54 км/ч 29.11.2025, 51 км/ч 06.07.2025), то:
  - Запись 1 (55 км/ч): history = [{54 км/ч, 29.11.2025}, {51 км/ч, 06.07.2025}]
  - Запись 2 (54 км/ч): history = [{55 км/ч, 11.05.2026}, {51 км/ч, 06.07.2025}]
  - Запись 3 (51 км/ч): history = [{55 км/ч, 11.05.2026}, {54 км/ч, 29.11.2025}]

### 6. Хранение в базе данных
- **Таблица**: `speed_records`
- **Поля**: breed, sex, name, speed_km_h, date, screenshot_url, status, history
- **Хранение истории**: JSON строка в поле `history`
- **Метод обновления**: `INSERT OR REPLACE` (инкрементальные обновления, без DELETE)
- **База данных**: D1 `pc-db` (remote)

### 7. API endpoint
- **Endpoint**: `/api/speed-records`
- **Метод**: GET
- **Лимит по умолчанию**: 1000 (query `limit`, макс. задаётся клиентом)
- **Формат ответа**: `{success: true, data: [...]}`

## Ключевые концепции

### Дедупликация vs История
- **Дедупликация**: Удаляет точные дубликаты (та же собака, та же дата, та же скорость) из разных листов
- **История**: Группирует разные записи для той же собаки (разные даты) для показа прогресса во времени

### Ключи группировки
- **Дедупликация**: `name + breed + sex + date + speed` (очень специфичный)
- **Формирование истории**: `name + breed` (шире, позволяет записи той же собаки с разным полом)
- **Отображение во фронтенде**: `name + breed` (одна запись на уникальную собаку, соответствует логике истории бэкенда)

### Общие проблемы

#### Проблема 1: В базе данных слишком много записей (2000+ вместо ~187)

**Симптомы:**
- База данных показывает 2000+ записей, когда должно быть ~187
- API возвращает дублирующиеся записи
- Фронтенд показывает дублирующихся собак

**Корневая причина:**
Множественные операции `INSERT OR REPLACE` без очистки старых данных. Каждый раз при запуске скрипта добавляются новые записи без удаления старых.

**Решение:**
```bash
# Шаг 1: Очистить базу данных
npx wrangler d1 execute pc-db --remote --command="DELETE FROM speed_records"

# Шаг 2: Перезагрузить данные
npx tsx backend/scripts/speed/fetch-speed-records.ts --remote

# Шаг 3: Проверить
npx wrangler d1 execute pc-db --remote --command="SELECT COUNT(*) as total FROM speed_records"
# Должно показать 187
```

**Профилактика:**
- Всегда очищайте базу данных перед крупными перезагрузками данных
- Рассмотрите добавление `DELETE FROM speed_records` в скрипт перед `INSERT OR REPLACE`
- Регулярно мониторьте размер базы данных

#### Проблема 2: Фронтенд показывает только одну запись на собаку, история не видна

**Симптомы:**
- Собака показывает только лучший результат
- Всплывающее окно истории не появляется при наведении
- Видно только 16-20 собак вместо 131

**Корневая причина:**
Ключ группировки фронтенда не соответствует ключу группировки истории бэкенда. Если бэкенд группирует по `name + breed`, а фронтенд по `name + breed + sex`, записи с разными значениями пола не будут сгруппированы вместе.

**Решение:**
Убедитесь, что бэкенд и фронтенд используют одинаковый ключ группировки:

**Бэкенд** (`backend/scripts/speed/fetch-speed-records.ts` строка 240):
```typescript
const key = `${record.name}_${record.breed}`;
```

**Фронтенд** (`frontend/src/pages/SpeedRecords/index.tsx` строка 105):
```typescript
const key = `${record.name}_${record.breed}`;
```

#### Проблема 3: Бесконечный цикл рендера React

**Симптомы:**
- Ошибка "Maximum update depth exceeded" в консоли
- Страница зависает или становится неотвечающей
- Высокое использование CPU

**Корневая причина:**
`setState` внутри `useCallback` или `useEffect` с неправильными зависимостями создаёт бесконечный цикл.

**Решение:**
Перестройте код, чтобы избежать setState в useCallback:

**До (проблематично):**
```typescript
const applyFilters = useCallback((baseRecords) => {
  // ... логика фильтрации
  setRecords(sorted); // setState в useCallback вызывает цикл
}, [filterYears, filterBreeds, filterSexes, searchQuery, sortField, sortDirection]);

useEffect(() => {
  applyFilters(allRecords);
}, [allRecords, applyFilters]); // applyFilters меняется при каждом рендере
```

**После (правильно):**
```typescript
const applyFilters = useCallback((baseRecords) => {
  // ... логика фильтрации
  return sorted; // Возвращаем данные вместо setState
}, [filterYears, filterBreeds, filterSexes, searchQuery, sortField, sortDirection]);

const filteredRecords = useMemo(() => {
  return applyFilters(allRecords);
}, [allRecords, applyFilters]);

useEffect(() => {
  setRecords(filteredRecords);
}, [filteredRecords]); // filteredRecords стабилен
```

#### Проблема 4: API / Stats возвращают не все записи

**Симптомы:**
- «Распределение скоростей» показывает ~100 замеров вместо полной базы
- Сумма по диапазонам не сходится с ожидаемым числом строк

**Корневая причина:**
`Stats.tsx` ранее вызывал `getSpeedRecords` с `limit=100`. Таблица использует `useSpeedRecordsPage` с `limit=1000`.

**Решение (актуально с 2026-07-06):**
- Данные грузятся в `useSpeedRecordsPage.ts` с `limit=1000` для обоих API (достаточно для текущей базы ~200 speed / ~100 coursing).
- Легаси-вью `stats/SpeedStatsView.tsx` / `CoursingStatsView.tsx` использовали `limit=10000` — в новом UI статистика в `DoninoStatsColumns.tsx` на тех же 1000 записях из page hook.
- При росте базы >1000 — увеличить limit в `useSpeedRecordsPage.ts` или добавить отдельный stats-endpoint.

**Проверка:**
```bash
curl "https://coursing-stats.ru/data/v1/donino/speed_records.json" | jq '.records | length'
```

#### Проблема 5: Отсутствует история для некоторых собак

**Симптомы:**
- У некоторых собак есть всплывающее окно истории при наведении
- У других собак история не показывается
- У собаки есть несколько записей в Google Sheets, но нет истории на сайте

**Корневая причина:**
1. Записи собаки имеют разные значения породы (например, "Уиппет" vs "Whippet")
2. Записи собаки имеют разные написания имени (например, "Тайга" vs "Тая")
3. Записи находятся в разных листах с небольшими вариациями

**Решение:**
1. Проверьте данные Google Sheets на консистентность
2. Убедитесь, что ключ группировки соответствует ожидаемому поведению
3. Проверьте, действительно ли записи находятся в разных листах

**Отладка:**
```bash
# Проверить конкретную собаку в базе данных
npx wrangler d1 execute pc-db --remote --command="SELECT * FROM speed_records WHERE name LIKE '%Тайга%'"

# Проверить логи дедупликации
npx tsx backend/scripts/speed/fetch-speed-records.ts
# Ищите строку "After deduplication"
```

#### Проблема 6: Видны дублирующиеся записи

**Симптомы:**
- Та же собака появляется несколько раз в таблице
- Та же запись появляется с разными ID
- Неконсистентные данные

**Корневая причина:**
Логика дедупликации работает некорректно или записи имеют небольшие вариации.

**Решение:**
1. Проверьте ключ дедупликации в `fetch-speed-records.ts`
2. Убедитесь, что дедупликация включает все необходимые поля: `name + breed + sex + date + speed_km_h`
3. Проверьте, имеют ли записи разные даты (разные записи, не дубликаты)

**Отладка:**
```bash
# Проверить дубликаты в базе данных
npx wrangler d1 execute pc-db --remote --command="SELECT name, breed, date, COUNT(*) as count FROM speed_records GROUP BY name, breed, date HAVING count > 1"
```
